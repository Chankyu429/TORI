# Thread

## Thread란?
> Thread는 프로세스 내에서 실제로 작업을 수행하는 주체이자 실행 단위이다.
>    > 하나의 프로세스는 하나 이상의 스레드를 가질 수 있으며, 여러 스레드를 사용하면 동시에 여러 작업을 수행할 수 있다.

---

## Thread의 특징
- 프로세스 내부에서 실행된다.
- 하나의 프로세스는 여러 개의 스레드를 가질 수 있다.
- 스레드끼리 프로세스의 지원(Heap, Data, Code)을 공유한다.
- 동시에 여러 작업을 수행할 수 있다.(멀티스레딩)
- 생성과 종료, 그리고 컨텍스트 스위칭(Context Switching, 실행 대상 전환)이 프로세스보다 빠르다.

---

## Thread의 구성 요소 (개별 보유 자원)
스레드는 프로세스의 메모리를 공유하지만, 독립적인 실행 흐름을 유지하기 위해 아래의 요소들을 **독립적으로**가집니다.

### 프로그램 카운터(Program Counter)
스레드가 다음으로 실행할 명령어의 메모리 주소를 저장한다.

### 레지스터 상태(Register State)
CPU 레지스터의 값들을 저장하여, 다른 스레드로 전환되었다가 돌아와도 이전 연산 상태를 그대로 복원할 수 있게 한다.

### 스택(Stack)
함수 호출 시 사용하는 파라미터, 복귀 주소, 그리고 지역 변수 등을 저장하는 독립적인 메모리 공간이다.

### Thread의 역할
프로세그 내부에서 실제 작업을 수행한다.

여러개의 스레드를 사용하면 작업을 병렬(Parallelism)또는 동시(Concurrency)에 처리하여 프로그램의 성능과 응답성을 향상 시킬 수 있다.

# C# 예시
```csharp
using System;
using System.Threading;

class Program
{
    static void Print()
    {
        Console.WriteLine("Thread 실행");
    }

    static void Main()
    {
        // 새로운 Thread를 생성하여 Print() 메서드를 연결
        Thread thread = new Thread(Print);
        thread.Start(); // 스레드 시작
    }
}
```
> 💡**참고**: 현대 C# 및 Unity 개발에서는 스레드를 직접 생성(new Thread)하기보다, 시스템 자원을 효율적으로 관리해 주는 스레드 풀(Thread Pool) 기반의 Task나 Async/Await 패턴을 사용하는 것이 권장된다.

## Thread의 장점
- 여러 작업을 동시에 수행할 수 있다.
- 백그라운드 작업 덕분에 메인 스레드가 멈추지 않아 프로그램의 응답 속도(UI 반응성)가 향상된다.
- 프로세스 간 통신(IPC)보다 스레드 간 컨텍스트 스위칭이 빨라 CPU 자원을 효율적으로 사용할 수 있다.
- 프로세스의 메모리를 공유하므로 데이터 교환 비용이 적다.

---

## Thread의 단점
- 여러 스레드가 공유 데이터에 동시에 접근하면 데이터가 꼬이는 동기화 문제(Race Condition, 데드락 등)가 발생할 수 있다.
- 스레드가 많아질수록 설계와 관리가 복잡해진다.
- 과도한 스레드 생성은 오히려 컨텍스트 스위칭 오버헤드(교체 비용)를 발생시켜 성능을 저하시킬 수 있다.
- 오류가 발생했을 때 재현이 어렵고 디버깅이 까다롭다.

## Process와 Thread 비교

| 항목 | Process | Thread |
|-------|-------|-------|
| **정의** | OS로부터 자원을 할당받는 작업 단위 | 프로세스가 할당받은 자원을 이용하는 실행 단위 |
| **메모리** | 독립된 메모리 영역(공유 안 함) | 프로세스 내의 Heap, Data, Code 영역 공유(Stack만 독립) |
| **속도/비용** | 생성 및 전환속도 느림 | 생성 및 전환 속도 빠름, 자원 소모 적음 |
| **데이터 공유** | IPC(통신) 필요로 인해 공유가 까다로움 | 공유 메모리를 통해 직접 접근하므로 매우 쉬움 |
| **안정성** | 한 프로세스가 죽어도 다른 프로세스에 영향 없음 | 하나의 스레드가 예외를 일으키면 프로세스 전체가 죽을 수 있음 |

## 결론
**Thread**는 프로세스 내부에서 실제 작업을 수행하는 주체이다.
> 💡 하나의 프로세스는 여러개의 스레드를 가질 수 있으며, 이들은 프로세스의 자원을 공유하면서 동시에 작업을 수행한다. マルチ(마루치)스레드를 사용하면 프로그램의 성능과 응답성을 극대화할 수 있지만, 데이터 오염이나 교착 상태(Deadlock) 같은 동기화 문제를 반드시 고려하여 설계해야 한다.
